# Self-Canonicalizing Spreadsheet Ingestion

Status: design note v0.6  
Scope: header-optional Excel/CSV ingestion for the normalized CDP registry  
Related DDL: `db/ddl/001-decision-registry-kernel.sql`  
Related contract: `docs/z-config-lookup-spreadsheet-ingestion.md`  
Related tests: `tests/test_self_canonicalizing_ingestion.py`  

---

## 1. Core principle

Human-authored spreadsheet headers are not authority.

The ingestion profile is authority.

The body of the spreadsheet is canonicalized into CDP-controlled columns.

```text
Human headers are hints.
Ingestion profiles are authority.
Canonical headers are generated by CDP.
```

This keeps attorneys, reviewers, and analysts out of data-cleansing work when the body of the spreadsheet is structurally right but the first row is missing, renamed, pretty, merged, or otherwise unhelpful.

Do not cleanse the attorney's spreadsheet.

Canonicalize it.

---

## 2. Why this exists

The normalized registry removed packed strings from the database:

```text
No key1/value1 pseudo-columns.
No actor_type:actor_id packed values.
No verb:object_type:object_id packed values.
No authoritative domain strings.
```

But a new operational risk appears at ingestion time:

```text
What happens if spreadsheet headers are wrong?
```

The answer should not be:

```text
Give the attorneys a data-cleansing project.
```

The answer should be:

```text
Use the ingestion profile to derive canonical headers by sheet role and column position.
```

---

## 3. Ingestion flow

The ingestion flow is:

```text
raw workbook / CSV
  -> detect ingestion profile
  -> identify sheet role
  -> ignore or preserve human headers as source metadata
  -> map columns by profile-defined ordinal position
  -> generate canonical headers
  -> normalize values
  -> validate identifiers, classes, and decisions
  -> insert canonical rows
  -> emit canonical workbook / CSV plus ingestion report
```

The input may have bad headers.

The output must have canonical headers.

---

## 4. Header handling rule

For v0.6, each sheet role has a fixed canonical column order.

If the first non-empty row looks like data, treat it as data.

If the first non-empty row does not match the expected row shape, treat it as a human header row and discard it for canonical ingestion.

In both cases, assign canonical column names by ordinal position.

This means the following are recoverable:

```text
missing header row
wrong header names
pretty labels
mixed capitalization
extra title words in headers
renamed Excel tabs
```

These are not recoverable and must fail validation:

```text
wrong column count
packed colon-delimited values in atomic fields
unknown actor IDs
mistyped actor IDs
unknown object IDs
mistyped object IDs
unknown permission sources
invalid parent-child relation types
invalid dates or booleans
```

Headers are presentation.

Values are governance.

---

## 5. Sheet-role detection

For a single workbook, ingestion should determine sheet roles using the following descending trust order:

1. explicit ingestion manifest or named range;
2. known sheet name;
3. expected sheet order;
4. expected column count;
5. row-shape inference.

For the first demo, the expected order is:

```text
1. identifier_registry
2. decision_class_registry
3. decision_registry
```

Sheet names may vary.

Column order may not.

## 7. Canonical artifacts

Ingestion should be able to emit clean artifacts after parsing:

```text
canonical_identifier_registry.csv
canonical_decision_class_registry.csv
canonical_decision_registry.csv
ingestion_report.json
ingestion_errors.csv
```

These canonical artifacts should contain CDP-generated headers, normalized booleans and dates, and only validated rows.

The original workbook remains source evidence.

The canonical artifacts become testable ingestion evidence.

---

## 8. Self-registering identifiers

There are two different ideas that must not be confused.

### 8.1 Self-canonicalizing columns

This is safe and should be automatic.

```text
Column 1 becomes registry_name.
Column 2 becomes decision_id.
Column 3 becomes decision_class_id.
```

### 8.2 Self-registering identifiers

This is powerful and must be controlled.

If a decision references:

```text
subject_actor_type = agent
subject_actor_id = claims_review_agent
```

and `claims_review_agent` is missing from `identifier_registry`, ingestion may produce a candidate identifier row.

For demo-only use, this may be allowed with:

```text
auto_register_identifiers = true
```

For production/legal use, prefer:

```text
auto_register_identifiers = proposed
```

That allows ingestion to generate candidate rows without silently making them authoritative.

The v0.6 unit tests focus on deterministic canonicalization and validation, not production auto-approval of new identifiers.

## 9. Unit-test contract

The unit tests must prove:

1. bad human headers are ignored;
2. headerless/body-first rows are accepted when the row shape is valid;
3. canonical headers are generated by profile;
4. Excel workbooks can be ingested by sheet role and ordinal column position;
5. CSV files can be ingested the same way;
6. missing identifiers fail;
7. mistyped identifiers fail;
8. packed colon-delimited fields fail before insert;
9. parent-child edges are derived from atomic IDs;
10. class rollups are derived from atomic class IDs.

The test fixture intentionally uses non-canonical human headers such as:

```text
Thingy, Decision #, Class-ish, Parent Maybe, Relationship, Why, Who Type, Who, Did What, Thing Type, Thing ID, Permission Kind, Permission, Needs Person?, Person, When
```

Ingestion must still produce:

```text
registry_name,decision_id,decision_class_id,parent_decision_id,parent_relation_type,antecedent_text,subject_actor_type,subject_actor_id,predicate_verb,object_type,object_id,permission_source_type,permission_source_id,human_required,human_approver_id,created
```

---

## 10. Design rule

Do not trust spreadsheet headers.

Do not infer governance from prose.

Do not parse packed strings from atomic columns.

Do not create floating IDs.

Use the ingestion profile to create canonical rows.

Use the identifier registry to validate references.

Use projections to create attorney-facing outputs.
