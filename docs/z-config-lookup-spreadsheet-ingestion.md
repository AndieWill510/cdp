# Spreadsheet Ingestion Contract for the Normalized Decision Registry

Status: Demo ingestion contract v0.6  
Scope: Header-optional, self-canonicalizing identifier/class/decision spreadsheet ingestion  
Audience: implementers, test authors, attorney-facing demo builders  
Related DDL: `db/ddl/001-decision-registry-kernel.sql`  
Related design notes: `docs/identifier-registry.md`, `docs/self-canonicalizing-ingestion.md`  
Related tests: `tests/test_self_canonicalizing_ingestion.py`  

---

## 1. Purpose

This document defines spreadsheet ingestion for the normalized CDP control-plane registry.

The canonical targets are:

```text
cdp_core.identifier_registry
cdp_core.decision_class_registry
cdp_core.decision_registry
```

The old `z_config_lookup` / key-value idea is not the canonical control-plane model.

The canonical model uses:

```text
atomic columns
profile-derived canonical headers
no-floating-ID identifier validation
```

Human-authored spreadsheet headers are not authoritative.

The ingestion profile is authoritative.

---

## 2. Header-optional rule

Spreadsheets may have:

```text
canonical headers
wrong headers
pretty headers
renamed headers
no headers
```

All are acceptable when the body is structurally valid.

Ingestion must assign canonical headers using:

```text
sheet role + canonical column order + row-shape validation
```

Human headers, if present, are source metadata only.

They must not decide database column names.

```text
Human headers are hints.
Ingestion profiles are authority.
Canonical headers are generated by CDP.
```

---

## 3. Ingestion order

Load spreadsheets in this order:

```text
1. identifier_registry seed rows
2. decision_class_registry rows
3. additional identifier_registry rows for actors, objects, policies, tools, etc.
4. decision_registry rows
5. projection queries
```

Why this order?

Because `decision_registry` rows are validated against `identifier_registry` and `decision_class_registry`.

A decision row that references an actor, object, predicate, permission source, source system, or relation type must not insert unless those identifiers already exist.

No floating IDs.

## 5. Sheet-role detection

For workbooks, detect sheet role in this order:

1. explicit ingestion manifest or named range;
2. known sheet name;
3. expected sheet order;
4. expected column count;
5. row-shape inference.

For the v0.6 demo workbook, the default sheet order is:

```text
1. identifier_registry
2. decision_class_registry
3. decision_registry
```

Sheet names may be human-friendly or wrong.

Column order may not be wrong.

---

## 6. Header detection

For each sheet:

1. remove completely empty rows;
2. inspect the first non-empty row;
3. if it matches the expected row shape, treat it as data;
4. if it does not match the expected row shape, treat it as a human header row;
5. assign canonical headers by ordinal position;
6. preserve the discarded human header only as source metadata.

Recoverable:

```text
Thingy, Decision #, Class-ish, Parent Maybe, Relationship, Why, Who Type, Who, Did What, Thing Type, Thing ID, Permission Kind, Permission, Needs Person?, Person, When
```

Not recoverable:

```text
wrong number of columns
packed values in atomic fields
missing identifiers
mistyped identifiers
invalid parent relation type
invalid date or Boolean values
```

## 7. Identifier registry rows

Identifier registry rows map into:

```text
cdp_core.identifier_registry
```

Canonical header:

```text
registry_name,identifier_id,identifier_type_registry_name,identifier_type_id,parent_registry_name,parent_identifier_id,display_label,description,status,created
```

Pairing rules:

```text
identifier_type_registry_name and identifier_type_id must both be blank or both populated.
parent_registry_name and parent_identifier_id must both be blank or both populated.
```

Decision validation accepts identifiers whose status is:

```text
active
deprecated
```

It rejects:

```text
inactive
retired
missing rows
```

## 8. Decision class rows

Decision class rows map into:

```text
cdp_core.decision_class_registry
```

Canonical header:

```text
registry_name,class_id,parent_class_id,class_label,class_level,created
```

Blank `parent_class_id` should be normalized to SQL `NULL`.

## 9. Decision rows

Decision rows map into:

```text
cdp_core.decision_registry
```

Canonical header:

```text
registry_name,decision_id,decision_class_id,parent_decision_id,parent_relation_type,antecedent_text,subject_actor_type,subject_actor_id,predicate_verb,object_type,object_id,permission_source_type,permission_source_id,human_required,human_approver_id,created
```

Decision validation rules:

1. `(registry_name, decision_class_id)` exists in `decision_class_registry`.
2. If `parent_relation_type = none`, `parent_decision_id` is blank/null.
3. If `parent_relation_type != none`, `parent_decision_id` is populated and references an existing decision in the same registry.
4. `parent_relation_type` exists in `identifier_registry` under `parent_relation_type`.
5. `subject_actor_type` exists in `identifier_registry` under `actor_type`.
6. `subject_actor_id` exists in `identifier_registry` under `actor` and is typed by `subject_actor_type`.
7. `predicate_verb` exists in `identifier_registry` under `predicate_verb`.
8. `object_type` exists in `identifier_registry` under `object_type`.
9. `object_id` exists in `identifier_registry` under `object` and is typed by `object_type`.
10. `permission_source_type` exists in `identifier_registry` under `permission_source_type`.
11. `permission_source_id` exists in `identifier_registry` under `permission_source` and is typed by `permission_source_type`.
12. `human_required` is normalized to Boolean.
13. `human_approver_id = none` is allowed when human approval is not required or no approver is recorded.
14. `human_approver_id = unknown` must exist as an unknown actor.
15. Any other `human_approver_id` must exist as an actor typed `human`.
16. `source_system` defaults to `spreadsheet` and must exist in `identifier_registry` under `source_system`.

The database trigger performs these checks at insert/update time.

The unit tests also perform these checks in memory for fixture validation.

## 10. Canonical artifacts

Ingestion should be able to emit:

```text
canonical_identifier_registry.csv
canonical_decision_class_registry.csv
canonical_decision_registry.csv
ingestion_report.json
ingestion_errors.csv
```

Canonical artifacts contain CDP-generated headers.

Original spreadsheet headers, if present, are evidence only.

---

## 11. Normal form rule

The v0.6 registry is normalized. It does not use:

```text
domain
key1/value1
key2/value2
key3/value3
actor_type:actor_id
verb:object_type:object_id
```

Compatibility strings such as:

```text
decision_register:<registry_name>:<decision_id>
decision_class:<registry_name>:<class_id>
identifier:<registry_name>:<identifier_id>
```

may appear in projection views.

They are not authoritative stored fields.

## 12. Analytics targets

The DDL exposes these read surfaces:

```text
cdp_projection.identifier_registry_flat
cdp_projection.decision_class_registry_flat
cdp_projection.decision_registry_flat
cdp_projection.decision_class_rollup
cdp_projection.decision_parent_child_edges
```

Use `decision_registry_flat` for attorney-facing review rows.

Use `decision_class_rollup` for class counts, human-review burden, and unknown-permission findings.

Use `decision_parent_child_edges` for lineage and graph/tree analysis.

Use `identifier_registry_flat` to inspect registered IDs and their types.

## 13. Unit-test contract

The tests in `tests/test_self_canonicalizing_ingestion.py` prove:

1. bad human headers are ignored;
2. canonical headers are derived by profile;
3. CSV fixture ingestion works;
4. Excel workbook ingestion works;
5. identifier rows load before class and decision rows;
6. missing identifiers fail;
7. mistyped identifiers fail;
8. packed fields fail before insert;
9. parent-child lineage is derived from atomic IDs;
10. class rollups are derived from atomic class IDs.

## 14. Final v0.6 contract

```text
Identifier registry:
  cdp_core.identifier_registry

Decision classes:
  cdp_core.decision_class_registry

Decisions:
  cdp_core.decision_registry

Generic config:
  cdp_core.config_lookup

Headers:
  optional, non-authoritative, generated by profile

No hidden magic.
No key/value pseudo-columns.
No packed colon-delimited facts in core tables.
No parent-child inference from prose.
No floating IDs.
No attorney data-cleansing project when the body is right enough.
```
